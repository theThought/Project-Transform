Accessibility considerations & other notes

  • All page content should be inside landmark regions, to assist screen reader users to quickly navigate content.
  • Getting question labelling right in a form is more important than headings - i.e providing an accessible name via <label>, <legend> or ARIA.
  • Correct heading structure is more important than sub-dividing <main> into further <section> and <article>, etc. This assists screen reader users with quickly identifying content hierarchy and providing an overview of the page.
  • Bear in mind that headings (or any editorial content) is only announced by screen readers in "browse mode".
    • If user is traversing interactive form controls using TAB key, they will be in "focus mode", so editorial content will be bypassed (unless it is programmatically associated with aria-describedby).
    • If the focus page property has "control":true, focus will be automatically placed inside the control, therefore the control's accessible name will be announced immediately.
    • See understanding screen reader modes.

Layout

  • Each question is sub-divided into 2 cols to test how semantic markup patterns work in 2-col layouts (e.g. <fieldset> with a <legend> as its 1st direct descendant).
  • Column layout uses Every Layout sidebar pattern.
  • For demo purposes, the 1st question uses a default width of 20% for col 1, and 66% for col 2 (i.e. col 2 cannot take up less than 66% of available width before the content wraps into a single column).
  • The 2nd question has inline override of CSS custom properties - 20rem fixed width for col 1, and 50% for col 2.
  • This col width override can be applied to an ancestor element, so that it affects ALL questions on the page.
  • Content overflow should never happen, unless an internal component has an explicit width set on it. We should avoid setting explicit widths on anything. Instead, we should use max-inline-size/max-width to constrain internal components if necessary.
